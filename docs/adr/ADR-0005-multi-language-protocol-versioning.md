# ADR-0005 多语言客户端协议版本策略

## 状态

Accepted

## 背景

`stellflow` 已明确采用以下通信路线：

- Broker / Client 数据面使用自定义二进制协议
- Java 客户端底层使用 Netty
- Golang 客户端底层自研实现

这意味着协议本身必须独立于具体语言运行时与网络框架，并且能够支持：

- Java 客户端实现
- Golang 客户端实现
- 后续更多语言客户端实现

因此必须提前明确协议版本管理、兼容策略与演进边界，避免后续不同语言客户端在协议升级过程中出现碎片化实现。

## 决策

`stellflow` 的多语言客户端协议版本策略采用“显式版本 + 能力协商 + 向后兼容优先”的原则：

1. 每个协议请求都必须包含显式 `apiKey` 与 `apiVersion`。
2. Broker 必须暴露协议版本查询能力，用于客户端启动后的能力探测。
3. 客户端必须基于 Broker 返回的能力范围选择可用版本，而不是假定服务端支持最新版本。
4. 协议演进优先采用向后兼容方式，避免无必要的破坏性变更。
5. 协议规范文档必须独立于 Java / Netty 实现存在。
6. Java、Golang 及后续其它语言客户端必须遵循同一份协议规范和错误码定义。

## 备选方案

### 方案 A：不做显式版本控制，协议随代码同步演进

优点：

- 初期实现简单

缺点：

- 一旦存在多个客户端版本，将快速失控
- 多语言客户端无法稳定追踪服务端变更
- 任何协议调整都可能导致整个生态同时升级

### 方案 B：显式版本控制，但不做能力协商

优点：

- 协议结构清晰

缺点：

- 客户端需要预先硬编码服务端能力
- 版本升级和回滚时适配成本高
- 不利于滚动升级场景

### 方案 C：显式版本 + 能力协商 + 向后兼容优先

优点：

- 最适合多语言客户端生态
- 有利于服务端滚动升级与灰度发布
- 有利于协议长期稳定演进

缺点：

- 协议设计与测试要求更高
- 需要为版本兼容保留更多实现分支

## 决策理由

最终选择方案 C，原因如下：

- `stellflow` 已经明确不是只服务 Java 单一客户端
- 自定义二进制协议一旦没有稳定版本机制，多语言实现成本会迅速上升
- 能力协商是实现 Broker 与客户端解耦升级的关键
- 该策略也最接近成熟消息中间件协议的长期演进方式

## 决策后果

采用该策略后：

- Broker 需要提供协议能力探测接口
- 客户端初始化流程必须包含版本协商
- 协议文档、测试用例和错误码表需要作为跨语言契约维护
- 服务端在引入新字段、新请求和新错误码时必须明确兼容策略

## 实现约束

- 不允许将协议格式直接绑定到 Netty 内部对象模型
- 不允许客户端默认假定服务端支持最新协议版本
- 新协议版本上线时必须说明兼容关系
- 所有语言 SDK 都必须基于统一协议契约测试
