# ADR-0006 元数据发现与 Broker 扩容策略

## 状态

Accepted

## 背景

`stellflow` 采用 Kafka 风格的分布式日志与分区路由模型，客户端不会一直通过单一接入点转发全部请求，而是需要：

- 先连接引导节点
- 再获取集群元数据
- 然后按分区 Leader 直连目标 Broker

在这种模型下，必须明确：

- 客户端如何发现集群
- Broker 地址如何暴露
- Broker 扩容和机房迁移时如何减少客户端配置变更

## 决策

`stellflow` 的元数据发现与 Broker 扩容策略如下：

1. 客户端通过一组 `bootstrap servers` 作为引导入口。
2. `bootstrap servers` 只负责引导，不代表后续所有业务流量都经过这些节点。
3. 客户端连接任一可达引导节点后，必须通过元数据请求获取完整集群视图。
4. Broker 对外暴露地址必须使用稳定的逻辑地址，例如域名，而不是依赖裸 IP。
5. 客户端后续按元数据返回的分区 Leader 地址直连目标 Broker。
6. Broker 扩容依赖控制面元数据更新与分区迁移，不依赖客户端手工重配全部连接目标。
7. 机房迁移场景下，应通过稳定域名和 DNS 或等效寻址机制减少客户端配置变更。

## 备选方案

### 方案 A：客户端固定连接单地址，由网关代理所有数据流量

优点：

- 客户端配置最简单

缺点：

- 不符合 Kafka 风格的数据面模型
- 网关可能成为吞吐瓶颈或故障热点
- 分区级路由控制力不足

### 方案 B：客户端写死 Broker IP 列表

优点：

- 实现简单

缺点：

- 扩容和机房迁移成本极高
- 地址变化会造成大面积客户端重配
- 不适合长期演进

### 方案 C：bootstrap + metadata + 稳定逻辑地址

优点：

- 与分布式日志系统模型天然匹配
- 更利于扩容、迁移与滚动变更
- 客户端和集群拓扑演进更解耦

缺点：

- 需要实现元数据缓存、刷新与失效处理
- 部署侧必须维护稳定寻址体系

## 决策理由

最终选择方案 C，原因如下：

- `stellflow` 的路由模型本身就要求客户端能够感知分区 Leader
- 稳定逻辑地址比裸 IP 更适合长期运维
- Broker 扩容本质上应通过元数据和分区迁移完成，而不是要求客户端重配全部节点

## 决策后果

采用该策略后：

- 客户端必须实现元数据拉取、缓存、刷新和失效重试逻辑
- Broker 元数据中返回的地址信息必须是可长期稳定管理的逻辑地址
- 扩容和迁移会依赖控制面与运维层的配合，而不是单靠客户端静态配置

## 实现约束

- 不将裸 IP 作为默认推荐的外部寻址方式
- `bootstrap servers` 不得被定义为唯一业务入口
- 客户端必须支持元数据刷新与 Leader 变更重试
- Broker 元数据地址设计必须与跨机房迁移策略兼容
