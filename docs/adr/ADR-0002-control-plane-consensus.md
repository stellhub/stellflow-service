# ADR-0002 控制面一致性选型

## 状态

Accepted

## 背景

`stellflow` 已经明确采用 Controller Quorum 作为控制面核心，因此必须确定：

- 控制面是否采用 Raft 路线
- 是自研一致性协议还是复用成熟实现

该决策会直接影响：

- 元数据日志写入方式
- Controller 选主与提交模型
- Broker 注册和分区元数据传播机制
- 后续容错、恢复与快照设计

## 决策

Controller Quorum 一致性使用 Apache 的 Raft 实现。

## 备选方案

### 方案 A：自研 Raft

优点：

- 可完全定制实现细节
- 外部依赖更少

缺点：

- 一致性协议实现、验证和调试成本极高
- 容易在日志复制、提交、快照、故障恢复等细节上出现长期隐患
- 当前项目阶段不适合将大量精力投入到协议轮子重造上

### 方案 B：采用 Apache 的 Raft 实现

优点：

- 复用成熟的一致性基础设施
- 显著降低实现复杂度
- 更适合当前阶段聚焦元数据模型、控制器状态机和 Broker 协作逻辑

缺点：

- 需要适配外部框架抽象
- 需要接受其部分运行模型与接口约束

### 方案 C：不采用 Raft 路线

优点：

- 可以自行定义更轻量的控制面机制

缺点：

- 与当前已确定的 Controller Quorum 路线冲突
- 很难保持和 Kafka 主线工程思想一致

## 决策理由

最终选择方案 B，原因如下：

- `stellflow` 的重点不是重写一个新的共识协议，而是重建 Kafka 风格的消息队列核心
- 控制面应尽快建立稳定、可用的提交与回放机制
- 使用 Apache 的 Raft 更符合“控制面复用成熟基础设施，数据面保留核心语义”的总体原则

## 决策后果

采用该方案后：

- 控制面元数据状态推进必须建立在 Apache Raft 的日志提交语义之上
- `MetadataLog`、`MetadataLoader`、`ClusterControlManager`、`ReplicationControlManager` 等模块需要围绕该 Raft 实现进行装配
- 项目需要补充快照、恢复、仲裁节点运维等适配设计

## 实现约束

- 不自研新的通用一致性协议
- 不在控制面旁路引入与 Raft 冲突的元数据提交机制
- 所有元数据变更必须先进入控制面日志，再驱动状态机回放
