# ADR-0004 可观测性选型

## 状态

Accepted

## 背景

Kafka 原生可观测性以 JMX 为主，但 `stellflow` 需要面向：

- 现代可观测平台
- 多语言客户端生态
- 更统一的指标、链路与日志接入方式

因此必须决定项目是否继续沿用 JMX-first 路线，还是直接采用现代标准。

## 决策

`stellflow` 可观测性统一采用 OpenTelemetry 标准，不再支持 JMX。

## 备选方案

### 方案 A：延续 JMX-first

优点：

- 与 Kafka 原生路线更接近
- 对 JVM 应用接入成本低

缺点：

- 对多语言生态不友好
- 需要额外转换层才能与现代观测平台更好集成
- 不适合作为新项目的一等标准接口

### 方案 B：JMX + Prometheus / OTel 转换

优点：

- 平滑兼容旧思路
- 改造成本相对较低

缺点：

- 仍然保留 JMX 作为基础暴露模型
- 指标语义与接入路径更复杂
- 系统内部标准不够统一

### 方案 C：OpenTelemetry-first

优点：

- 指标、链路与日志体系更统一
- 对多语言生态更友好
- 更适合现代云原生与平台化接入

缺点：

- 与 Kafka 原生可观测性路径不同
- 需要在实现初期就明确 OTel 语义模型

## 决策理由

最终选择方案 C，原因如下：

- `stellflow` 是新项目，没有必要继续沿用 JMX 作为一等标准接口
- OTel 更符合未来多语言客户端和现代运维体系的接入方式
- 相比 JMX-first，OTel-first 更适合统一服务端指标、追踪和日志相关扩展

## 决策后果

采用该方案后：

- Broker、Controller、客户端 SDK 的可观测性设计应优先围绕 OTel 展开
- Prometheus 若作为后端之一，应通过 OTel 生态链路接入
- 不再提供 JMX 作为正式支持的主暴露方式

## 实现约束

- 不新增 JMX 作为主指标接口
- 指标命名、标签设计和追踪语义需尽早统一
- 服务端与客户端的可观测埋点优先复用 OpenTelemetry 标准模型
