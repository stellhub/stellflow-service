# ADR-0001 通信协议选型

## 状态

Accepted

## 背景

`stellflow` 需要同时解决三类通信问题：

- Broker / Client 数据面通信
- Broker 间数据复制通信
- Controller / Broker 控制面通信

项目整体目标是保持 Kafka 风格的核心语义与工程思想，但不要求在所有基础设施层面完全复刻 Kafka 原始实现路径。因此需要分别判断哪些链路应继续采用 Kafka 风格，哪些链路可以引入更现代的通信框架。

## 决策

本项目采用分层通信方案：

1. Broker / Client 通信：使用自定义二进制协议。
2. Java 客户端底层使用 Netty。
3. Golang 客户端底层自研实现。
4. Broker 间数据复制：复用同一套二进制协议。
5. Broker 间数据复制底层统一使用 Netty。
6. Controller / Broker 控制面通信：使用 gRPC。

## 备选方案

### 方案 A：所有链路统一使用 gRPC

优点：

- 多语言代码生成方便
- IDL 明确
- 控制面开发效率高

缺点：

- 数据面更像 RPC 系统，不利于保持 Kafka 风格的批量拉取、长轮询与二进制请求响应模型
- Broker / Client 热路径控制力下降
- 复制链路语义映射不够自然

### 方案 B：所有链路统一使用自定义二进制协议

优点：

- 语义最接近 Kafka
- 数据面与复制链路最统一

缺点：

- 控制面开发成本偏高
- 心跳、注册、配置变更等接口的演进成本较高
- 多语言控制面工具支持成本增加

### 方案 C：分层方案

优点：

- 数据面保留 Kafka 风格
- 控制面获得更高的演进效率和跨语言友好性
- 兼顾性能控制力与工程效率

缺点：

- 仓库内会同时存在两种通信技术路线
- 需要清晰管理数据面和控制面的边界

## 决策理由

最终选择方案 C，原因如下：

- Broker / Client 与复制链路属于热路径，应优先保持 Kafka 风格的二进制协议模型
- Controller / Broker 通信更偏管理与控制命令，gRPC 更适合表达这类接口
- Java 侧使用 Netty 可以减少底层 I/O 实现成本
- 协议本身独立于 Netty，保证 Golang 等语言可以独立实现客户端

## 决策后果

采用该方案后，项目会形成两套清晰分层：

- 数据面：自定义二进制协议
- 控制面：gRPC

这会带来以下结果：

- 数据面协议必须有稳定、清晰、跨语言可实现的规范
- Java 和 Golang 客户端的底层实现不同，但必须遵守同一协议语义
- 控制面接口会更容易维护和演进

## 实现约束

- 不将 gRPC 引入 Broker / Client 数据面主链路
- 不单独再定义一套新的 Broker 间复制协议
- 二进制协议设计必须独立于 Netty 实现
- 控制面接口优先围绕 gRPC IDL 设计与演化
