# ADR-0003 存储核心选型

## 状态

Accepted

## 背景

`stellflow` 的核心目标之一是保持 Kafka 风格的分布式日志语义，因此必须明确消息存储核心是否：

- 使用自研日志段
- 引入第三方 KV
- 通过 gRPC 做远程抽象

该决策会影响消息写入模型、索引结构、恢复逻辑、复制路径以及整体性能优化方向。

## 决策

消息存储核心采用自研日志段模型：

- 自研 `UnifiedLog + LogSegment + OffsetIndex + TimeIndex`
- 不走 gRPC
- 不走第三方 KV

## 备选方案

### 方案 A：自研日志段与索引

优点：

- 与 Kafka 风格最一致
- 更适合顺序追加、批量读取、基于 offset 的寻址模型
- 可以精确控制恢复、截断、滚段和复制语义

缺点：

- 存储层实现工作量较大
- 需要自行处理文件、索引、恢复和清理逻辑

### 方案 B：引入 RocksDB / LevelDB 等 KV

优点：

- 可复用现成持久化能力
- 某些状态存储场景开发较快

缺点：

- 核心模型会偏离日志追加语义
- 对高吞吐顺序写和稀疏索引场景不天然匹配
- 会让消息系统更像“KV 驱动的消息层”，而不是“日志驱动的消息层”

### 方案 C：将存储抽象为 gRPC 远程服务

优点：

- 存储层可独立部署与演进

缺点：

- 热路径增加远程调用成本
- 与本地日志事实模型冲突
- 不符合当前项目阶段与目标

## 决策理由

最终选择方案 A，原因如下：

- Kafka 风格消息队列的核心不是 KV，而是顺序追加日志
- 数据面热路径必须避免额外远程抽象
- 复制、截断、恢复和高水位推进都天然依赖日志段模型

## 决策后果

采用该方案后：

- 存储层将成为项目中最重要的自研核心之一
- 需要尽早稳定文件格式、索引结构和恢复语义
- 辅助状态未来即便引入第三方引擎，也不得侵入主消息存储路径

## 实现约束

- 不将第三方 KV 用作主消息存储
- 不把 gRPC 引入主消息写读路径
- 复制、恢复、截断等语义必须围绕日志段模型设计
