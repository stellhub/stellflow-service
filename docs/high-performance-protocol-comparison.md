# Stellflow 高性能协议选型与吞吐设计对照

## 1. 文档目的

本文档用于对比当前主流高性能通信协议或通信协议栈的特征，并结合 `stellflow` 的目标，给出协议选型建议与高吞吐数据面设计原则。

本文档重点回答以下问题：

- HTTP、gRPC 等协议的头部开销大致是什么特点
- 当前常见高性能协议分别适合什么场景
- 它们的协议设计特征是什么
- `stellflow` 这种以大吞吐消息传输为目标的系统，为什么应当采用当前已确定的协议分层
- 如果目标是 `500 MB/s+` 的数据面吞吐，应优先优化哪些点

## 2. 核心结论

对于 `stellflow` 这种以高吞吐、批量传输、顺序日志写入和副本复制为核心的系统，最重要的不是单纯追求“最小协议头”，而是保证以下能力：

- 二进制协议
- 批量传输
- 多路复用或 pipeline
- 少拷贝或零拷贝
- 顺序 I/O
- 清晰的流控与错误模型
- 数据面和控制面分离

因此：

- Broker / Client 数据面最适合自定义二进制协议
- Broker 间复制最适合复用同一套二进制协议
- Controller / Broker 控制面最适合使用 gRPC

## 3. 协议头与帧开销对比

### 3.1 HTTP/1.1

特点：

- 文本协议
- 没有固定总头大小
- 头部大小取决于 URL、Header 数量、认证信息、Cookie、Trace 信息等

结论：

- 在高频小请求下，头部冗余明显
- 不适合作为高吞吐消息数据面协议

### 3.2 HTTP/2

特点：

- 二进制分帧
- 每个 frame 固定有 `9 bytes` frame header
- Header block 通过 `HPACK` 压缩
- 没有固定总 Header 大小

结论：

- 比 HTTP/1.1 更适合作为高性能 RPC 底层
- 但对超高吞吐消息数据面而言，仍然比自定义协议更重

### 3.3 gRPC

特点：

- 运行在 HTTP/2 之上
- 每条消息额外有 `5 bytes` message prefix
  - `1 byte` compression flag
  - `4 bytes` message length
- 同时还叠加 HTTP/2 HEADERS / TRAILERS 和 metadata

结论：

- 更适合控制面和通用 RPC
- 不适合作为 `stellflow` 数据面核心协议

### 3.4 自定义二进制协议

特点：

- 头部大小完全可控
- 可以设计成固定头 + 可变体
- 可以让消息体直接承载批量字节块

结论：

- 最适合吞吐优先的数据面设计

### 3.5 QUIC / HTTP/3

特点：

- 跑在 QUIC 之上
- 头部不是固定一个简单值
- 更适合公网、移动网络、弱网和高丢包环境

结论：

- 对公网服务很有价值
- 对当前 `stellflow` 这类内网消息系统，不是首选数据面协议

## 4. 主流高性能协议 / 协议栈对照表

| 协议或协议栈 | 性能定位 | 典型特点 | 优点 | 缺点 | 更适合的场景 |
| --- | --- | --- | --- | --- | --- |
| 自定义二进制协议 + TCP | 吞吐和控制力最高 | 头部最小化、批量传输、可零拷贝、语义强贴合业务 | 性能上限高、最适合顺序日志系统 | 多语言 SDK 成本高 | MQ 数据面、日志复制、文件块传输 |
| gRPC over HTTP/2 | 高性能通用 RPC | Protobuf、HTTP/2、多路复用、streaming | 多语言友好、IDL 清晰、开发效率高 | 比自定义协议更重 | 控制面、内部服务调用 |
| HTTP/2 | 高性能通用传输 | 二进制分帧、HPACK、多路复用 | 比 HTTP/1.1 高效很多 | 仍偏通用协议 | API、网关、RPC 底层 |
| HTTP/3 / QUIC | 高性能公网协议 | QUIC、多路复用、弱网友好 | 公网体验更好 | 内网高吞吐未必更优 | 跨公网访问 |
| WebSocket(binary) | 中高性能实时通信 | 长连接、双工、轻量帧 | 浏览器友好 | 缺少完整消息中间件语义 | 浏览器实时推送 |
| Aeron | 极致低延迟高吞吐 | UDP/IPC、direct buffer、低抖动 | 性能极强 | 专用性强、生态窄 | 超高性能内部传输 |
| ZeroMQ / ZMTP | 高性能消息传输 | 模式丰富、消息导向 | 灵活、高性能 | 不适合直接做标准业务协议 | 内部消息拓扑 |

## 5. 这些高性能协议的共同设计特点

真正高性能的协议或协议栈，通常都具备以下共同特征：

### 5.1 二进制化

- 避免文本解析成本
- 避免重复传输冗余字段名
- 降低序列化和反序列化 CPU 开销

### 5.2 批量化

- 单次传输尽量承载更大数据块
- 避免每条消息都触发一次完整协议处理链
- 更容易逼近磁盘和网卡顺序传输效率

### 5.3 多路复用或 Pipeline

- 减少 RTT 对吞吐的限制
- 允许多个请求在飞
- 避免严格“一问一答”的串行瓶颈

### 5.4 少拷贝 / 零拷贝

- 减少用户态和内核态之间的复制次数
- 减少重复 buffer 拼接与对象分配
- 更容易打满网卡和磁盘吞吐

### 5.5 顺序 I/O 友好

- 高吞吐系统一般都依赖顺序写和顺序读
- 协议与存储模型要能天然配合连续字节块传输

### 5.6 明确的流控和错误模型

- 对请求大小、批量大小、节流、超时要有明确约束
- 对错误码要有稳定语义

## 6. Stellflow 的协议选型落点

结合当前 ADR 和设计文档，`stellflow` 的协议分层如下：

### 6.1 Broker / Client 数据面

- 使用自定义二进制协议
- Java 客户端底层使用 Netty
- Golang 客户端底层自研实现

原因：

- 数据面最重视吞吐、批量和顺序传输
- 需要最大限度贴合 `Produce / Fetch / RecordBatch` 语义

### 6.2 Broker 间数据复制

- 复用同一套二进制协议
- 底层统一使用 Netty

原因：

- 复制本质上也是高吞吐批量传输
- 不应再单独维护一套完全不同的复制协议

### 6.3 Controller / Broker 控制面

- 使用 gRPC

原因：

- 控制面消息频率低
- 更关注开发效率、跨语言接口一致性和演进便利性

## 7. 为什么数据面不选 gRPC

虽然 gRPC 本身性能并不差，但它并不是 `stellflow` 数据面的最佳选择。主要原因如下：

- 它更偏通用 RPC，而不是顺序日志系统的数据面语义
- `Produce / Fetch / Replica Fetch` 更适合批量二进制请求-响应模型
- 引入 HTTP/2 头、metadata、trailers 和 RPC 语义后，控制力下降
- 在超高吞吐目标下，不如自定义协议容易做极致优化

## 8. 500 MB/s+ 吞吐目标下的设计要点

### 8.1 批量是第一原则

如果目标是 `500 MB/s+`，必须保证消息不是按“单条”传输，而是按 `RecordBatch` 传输：

- Producer 先聚合再发送
- Broker 按批存储
- Consumer 按批拉取
- Replica 按批同步

### 8.2 保证 Broker 像日志批处理器，而不是逐条 RPC 处理器

Broker 应尽量做：

- 批量校验
- 批量位点分配
- 批量追加
- 批量返回

Broker 应尽量避免：

- 每条消息单独编码
- 每条消息单独 ACK
- 中途反复拆批再重组

### 8.3 减少内存拷贝

Java / Netty 侧建议：

- 使用 pooled direct buffer
- 优先 direct memory
- 利用 gather write
- 文件发送优先零拷贝路径

### 8.4 顺序日志存储

高吞吐数据面必须建立在：

- 顺序追加日志
- 稀疏索引
- 批量刷盘
- 批量读取

而不是：

- 随机 KV put/get
- 每条消息独立随机写

### 8.5 支持 in-flight 和 pipeline

- 一个连接上不能只发一个请求等一个响应
- Producer、Fetch、Replica Fetch 都应支持多请求在飞
- 要尽量减少 RTT 成为吞吐瓶颈

### 8.6 多分区并行扩吞吐

单连接或单分区总会有上限。要稳定冲高吞吐，必须通过多分区并行放大：

- 多个分区并发写入
- 多个复制流并发同步
- 多线程或多 event loop 并发处理

### 8.7 控制面与数据面彻底分离

如果想让数据面接近文件传输效率，就不能把控制面负担压到同一套重协议上。  
因此当前 `stellflow` 的分层是合理的：

- 控制面：gRPC
- 数据面：自定义二进制协议

## 9. 会显著拖累吞吐的典型做法

以下做法会明显降低吞吐：

- 用 HTTP/REST 做 Broker 数据面
- 用全量 gRPC 做 Broker 数据面
- 每条消息单独 ACK
- 每条消息单独压缩或校验
- Broker 中途反复拆包重组
- 数据面主链路走 KV
- 不支持 pipeline，只做串行请求-响应
- 过度依赖网关转发而不是按分区直连目标 Broker

## 10. 对 Stellflow 的最终建议

结合当前项目目标，协议选型建议如下：

1. Broker / Client：自定义二进制协议
2. Broker 间复制：复用同一套二进制协议
3. Controller / Broker：gRPC
4. 消息存储：顺序日志段模型
5. 高吞吐优化方向：批量、少拷贝、顺序 I/O、pipeline、多分区并行

## 11. 结论

对于 `stellflow` 而言，真正决定数据面高性能上限的，不是单纯比较“HTTP 头多大、gRPC 头多大”，而是要保证整条链路具备：

- 批量传输
- 二进制协议
- 少拷贝
- 顺序日志 I/O
- 并行 pipeline

也正因为如此，`stellflow` 当前已经确定的数据面协议路线，是最符合其高吞吐目标的方案。
